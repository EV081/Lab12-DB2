# P5. Algoritmos distribuidos en tres servidores

## Arquitectura Master-Worker

| Servidor | Contenedor | Puerto host | Fragmentos alojados |
|----------|------------|-------------|---------------------|
| Master (coordinador) | `pg_master` | 5432 | `AtencionMedica_Diabetes`, `Pacientes_F1` |
| Worker 1 | `pg_worker1` | 5433 | `AtencionMedica_Obesidad`, `AtencionMedica_Cardiopatia`, `Pacientes_F2` |
| Worker 2 | `pg_worker2` | 5434 | `AtencionMedica_Hipertension`, `Pacientes_F3` |

La fragmentación de `Pacientes` respeta el vector `["H","P"]` del enunciado:
- **F1** (Master): ciudades `< 'H'` → Arequipa, Ayacucho, Callao, Chiclayo, Chimbote, Cusco
- **F2** (Worker1): `'H' ≤ ciudad < 'P'` → Huancayo, Huaraz, Ica, Iquitos, Juliaca, Lima, Moquegua
- **F3** (Worker2): ciudades `≥ 'P'` → Piura, Pucallpa, Tacna, Trujillo, Tumbes

> **Todo se ejecuta con Docker.** No necesitas `psql` instalado localmente.  
> Usa `docker exec` para conectarte a cada contenedor y correr SQL.

---

## Paso 0: Crear `docker-compose.yml`

Crea el archivo `docker-compose.yml` en tu directorio de trabajo:

```yaml
services:
  pg_master:
    image: postgres:16
    container_name: pg_master
    environment:
      POSTGRES_PASSWORD: pass
    ports:
      - "5432:5432"
    networks:
      - pg_net

  pg_worker1:
    image: postgres:16
    container_name: pg_worker1
    environment:
      POSTGRES_PASSWORD: pass
    ports:
      - "5433:5432"
    networks:
      - pg_net

  pg_worker2:
    image: postgres:16
    container_name: pg_worker2
    environment:
      POSTGRES_PASSWORD: pass
    ports:
      - "5434:5432"
    networks:
      - pg_net

networks:
  pg_net:
    driver: bridge
```

## Paso 5: Configurar `postgres_fdw` en el Master

```bash

CREATE EXTENSION IF NOT EXISTS postgres_fdw;

-- Registrar Worker1 usando su nombre en la red Docker pg_net
CREATE SERVER worker1
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'pg_worker1', port '5432', dbname 'postgres');

-- Registrar Worker2
CREATE SERVER worker2
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'pg_worker2', port '5432', dbname 'postgres');
```

## Paso 7: Queries distribuidas (ejecutar en el Master)

Todas las queries se ejecutan desde el Master vía `docker exec`.

### Q1. SELECT * FROM Pacientes ORDER BY FechaNacimiento

```bash
docker exec -i pg_master psql -U postgres << 'EOF'
BEGIN;

-- R1: ordena localmente en Master (F1 — ciudades < 'H')
CREATE TEMP TABLE r1 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F1 ORDER BY FechaNacimiento;

-- R2: ORDER BY delegado a Worker1 vía FDW (F2 — 'H' a 'P')
CREATE TEMP TABLE r2 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F2_fdw ORDER BY FechaNacimiento;

-- R3: ORDER BY delegado a Worker2 vía FDW (F3 — >= 'P')
CREATE TEMP TABLE r3 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F3_fdw ORDER BY FechaNacimiento;

-- Merge final (k-way merge) en el coordinador
EXPLAIN ANALYZE
SELECT * FROM (
    SELECT * FROM r1
    UNION ALL SELECT * FROM r2
    UNION ALL SELECT * FROM r3
) m
ORDER BY FechaNacimiento;

COMMIT;
EOF
```

### Q2. SELECT DISTINCT CiudadOrigen FROM Pacientes

```bash
docker exec -i pg_master psql -U postgres << 'EOF'
BEGIN;

CREATE TEMP TABLE r1 ON COMMIT DROP AS
    SELECT DISTINCT CiudadOrigen FROM Pacientes_F1;

CREATE TEMP TABLE r2 ON COMMIT DROP AS
    SELECT DISTINCT CiudadOrigen FROM Pacientes_F2_fdw;

CREATE TEMP TABLE r3 ON COMMIT DROP AS
    SELECT DISTINCT CiudadOrigen FROM Pacientes_F3_fdw;

-- Sin duplicados entre fragmentos por diseño de la partición por rango
EXPLAIN ANALYZE
SELECT CiudadOrigen FROM r1
UNION ALL SELECT CiudadOrigen FROM r2
UNION ALL SELECT CiudadOrigen FROM r3;

COMMIT;
EOF
```

### Q3. SELECT Diagnostico, AVG(Edad) GROUP BY Diagnostico

```bash
docker exec -i pg_master psql -U postgres << 'EOF'
BEGIN;

CREATE TEMP TABLE r1 ON COMMIT DROP AS
    SELECT Diagnostico, AVG(Edad) AS PromEdad
    FROM AtencionMedica_Diabetes GROUP BY Diagnostico;

CREATE TEMP TABLE r2 ON COMMIT DROP AS
    SELECT Diagnostico, AVG(Edad) AS PromEdad
    FROM AtencionMedica_Obesidad_fdw GROUP BY Diagnostico;

CREATE TEMP TABLE r3 ON COMMIT DROP AS
    SELECT Diagnostico, AVG(Edad) AS PromEdad
    FROM AtencionMedica_Cardiopatia_fdw GROUP BY Diagnostico;

CREATE TEMP TABLE r4 ON COMMIT DROP AS
    SELECT Diagnostico, AVG(Edad) AS PromEdad
    FROM AtencionMedica_Hipertension_fdw GROUP BY Diagnostico;

-- 4 filas finales: cada fragmento calculó su AVG localmente
EXPLAIN ANALYZE
SELECT * FROM r1
UNION ALL SELECT * FROM r2
UNION ALL SELECT * FROM r3
UNION ALL SELECT * FROM r4;

COMMIT;
EOF
```

### Q4. SELECT * FROM Pacientes NATURAL JOIN AtencionMedica

```bash
docker exec -i pg_master psql -U postgres << 'EOF'
BEGIN;

-- Fase 1: proyección local de DNI en cada fragmento de AtencionMedica
CREATE TEMP TABLE dni_a1 ON COMMIT DROP AS
    SELECT DISTINCT DNI FROM AtencionMedica_Diabetes;

CREATE TEMP TABLE dni_a2 ON COMMIT DROP AS
    SELECT DISTINCT DNI FROM AtencionMedica_Obesidad_fdw;

CREATE TEMP TABLE dni_a3 ON COMMIT DROP AS
    SELECT DISTINCT DNI FROM AtencionMedica_Cardiopatia_fdw;

CREATE TEMP TABLE dni_a4 ON COMMIT DROP AS
    SELECT DISTINCT DNI FROM AtencionMedica_Hipertension_fdw;

-- Fase 2: unión deduplicada de DNIs en el coordinador
CREATE TEMP TABLE dni_atencion ON COMMIT DROP AS
    SELECT DNI FROM dni_a1
    UNION SELECT DNI FROM dni_a2
    UNION SELECT DNI FROM dni_a3
    UNION SELECT DNI FROM dni_a4;

-- Fase 3: semi-join en cada fragmento de Pacientes
-- FDW hace push-down del filtro IN al worker — evita traer la tabla entera
CREATE TEMP TABLE p_filt_1 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F1
    WHERE DNI IN (SELECT DNI FROM dni_atencion);

CREATE TEMP TABLE p_filt_2 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F2_fdw
    WHERE DNI IN (SELECT DNI FROM dni_atencion);

CREATE TEMP TABLE p_filt_3 ON COMMIT DROP AS
    SELECT * FROM Pacientes_F3_fdw
    WHERE DNI IN (SELECT DNI FROM dni_atencion);

-- Fase 4: reunir el lado izquierdo ya reducido en el coordinador
CREATE TEMP TABLE p_reduced ON COMMIT DROP AS
    SELECT * FROM p_filt_1
    UNION ALL SELECT * FROM p_filt_2
    UNION ALL SELECT * FROM p_filt_3;

-- Fase 5: consolidar AtencionMedica completa en el coordinador
CREATE TEMP TABLE atencion_completa ON COMMIT DROP AS
    SELECT * FROM AtencionMedica_Diabetes
    UNION ALL SELECT * FROM AtencionMedica_Obesidad_fdw
    UNION ALL SELECT * FROM AtencionMedica_Cardiopatia_fdw
    UNION ALL SELECT * FROM AtencionMedica_Hipertension_fdw;

-- Join final en el coordinador
EXPLAIN ANALYZE
SELECT * FROM p_reduced JOIN atencion_completa USING (DNI);

COMMIT;
EOF
```

## Explicación de resultados esperados

### Q1 y Q2 — Foreign Scan visible en EXPLAIN

El plan muestra `Foreign Scan on pacientes_f2_fdw` y `Foreign Scan on pacientes_f3_fdw`, evidenciando que la consulta fue ejecutada en los workers. Para Q1, FDW hace push-down del `ORDER BY`: el worker ordena localmente y envía datos ya ordenados, reduciendo el trabajo del merge en el master.

### Q3 — Agregación remota

Cada `AVG(Edad)` se calcula en el servidor que aloja ese fragmento. El master recibe solo 4 filas. El plan muestra `Foreign Scan on atencionmedica_obesidad_fdw` con costos mayores que un `Seq Scan` local, reflejando la latencia de red incluida en la estimación.

### Q4 — Semi-join con push-down de predicado

El filtro `WHERE DNI IN (...)` viaja al worker como condición remota. El `EXPLAIN` muestra la línea:
```
Remote SQL: SELECT ... FROM pacientes_f2 WHERE (dni = ANY ($1::bpchar[]))
```
Esto confirma que el worker filtra localmente y solo retorna las filas relevantes, no la tabla entera.

### Diferencia clave respecto a P4

| Nodo en EXPLAIN | P4 (local) | P5 (distribuido) |
|-----------------|-----------|-----------------|
| Lectura de fragmento | `Seq Scan on pacientes_f2` | `Foreign Scan on pacientes_f2_fdw` |
| Filtros | Aplicados en master | Push-down al worker vía FDW |
| Costo estimado | Bajo (memoria local) | Mayor (incluye latencia TCP) |
| Evidencia de distribución | Ninguna | Nodo `Foreign Scan` + `Remote SQL` |
